# Companion Behavior Engine Standard

## Purpose

This document defines how Spark Estate™ behavior must be designed, implemented, and verified.

The platform does not discover its intended behavior from scattered code.

The product standard defines the behavior. Code must implement it.

## Governing Sequence

**Situation → required answer → expected outcome → likely ADHD friction → permitted support → coded behavior → visible proof**

## Every Behavior Must Define

1. Triggering situation
2. User's likely expectation
3. Required platform answer
4. Required behavior
5. Prohibited behavior
6. Likely hidden ADHD friction
7. Friction-reduction behavior
8. Required information
9. Safe defaults
10. Decision rules
11. Permission rules
12. State requirements
13. Action requirements
14. Response requirements
15. Failure behavior
16. Regression tests
17. Live proof
18. Definition of success

## Governing Rules

- Help first. Learn gradually.
- Ask only what is materially necessary.
- Anticipate friction, not permission.
- Emotion changes how support is delivered, not the user's actual task.
- Current conversation outranks unrelated history.
- Explicit requests outrank inferred needs.
- One authoritative turn decision governs every turn.
- One trusted conversation state is used across the platform.
- One final response composer owns user-facing wording.
- Features and intelligence libraries advise; they do not independently control the conversation.
- A capability is incomplete until its visible behavior passes live verification.

## Beyond Expectations

Spark Estate™ should first satisfy the stated need.

Then, when useful and permitted, it should reduce friction the person may not have named by:

- preserving information
- reducing choices
- preparing the next step
- remembering what comes next
- making return easier
- completing work inside the conversation
- preventing avoidable repetition

Beyond expectations must never mean surprising action, excessive coaching, or taking control away.
