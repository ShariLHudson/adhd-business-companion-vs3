# Friction and Decision Fatigue Standard

## Purpose

Every Spark Estate™ interaction must reduce or avoid unnecessary burden.

## Friction Categories

### Cognitive Load
How much information must the person process?

### Decision Fatigue
How many choices must the person make before receiving help?

### Memory Burden
What must the person remember, relocate, or repeat?

### Initiation Friction
How hard is it to begin?

### Navigation Friction
How many screens, menus, or room changes are required?

### Setup Friction
What must be configured before the platform becomes useful?

### Emotional Friction
Does the experience create shame, pressure, uncertainty, or fear of doing it wrong?

### Recovery Friction
How hard is it to resume after interruption, absence, or failure?

## Required Design Rules

- One obvious primary action.
- No more choices than necessary.
- Progressive disclosure instead of large menus.
- Use ordinary language rather than platform terminology.
- Prefill known information.
- Preserve context and source material automatically.
- Do work in the conversation when possible.
- Never ask the person to repeat confirmed information.
- Never require profile completion for basic help.
- Default safely when missing information is not material.
- Ask one concise question when missing information is material.
- Offer alternatives without making the user evaluate all alternatives.
- Avoid making an overwhelmed person diagnose the type of overwhelm before receiving support.

## Friction Budget

Every new screen, question, choice, workflow transition, and setting must justify its burden.

If the same outcome can be achieved with fewer decisions or less memory, the lower-friction design is required.

## Release Check

A behavior fails release review when it adds avoidable:

- questions
- choices
- navigation
- repeated information
- setup
- interpretation work
- recovery work
