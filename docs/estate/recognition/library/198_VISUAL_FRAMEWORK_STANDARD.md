## Purpose

Define how every visual map is structured, customized, and reused.

A map type defines the way of thinking.

A framework defines the sections that belong inside that map.

---

## Core Principle

Every map starts with a smart default framework, but the member may change it.

Spark provides structure.

The member owns the structure.

---

## Framework Components

A framework may include:

- Title
- Purpose
- Sections
- Required questions
- Optional questions
- Node types
- Relationship types
- Suggested actions
- Save destinations
- Transformation options

## Editable Framework

Members may:

- Add sections
- Delete sections
- Rename sections
- Reorder sections
- Mark sections as important
- Hide sections
- Restore default sections
- Save a custom framework

---

## Spark Suggestions

Spark may suggest framework improvements.

Examples:

- "Would it help to add a Risk section?"
- "This looks like it needs a Timeline section."
- "You often include Budget in these maps. Add it?"
- "I think Stakeholders may be missing."

Spark must ask before changing the structure.

---

## Saved Frameworks

Members may save personal frameworks.

Examples:

- Shari's Launch Decision Map
- Workshop Planning Map
- Facebook Marketing Decision
- Client Journey Framework
- Monthly Strategy Map

---

## Learning Behavior

Spark may learn:

- Sections the member often adds
- Sections the member usually removes
- Preferred map types
- Favorite frameworks
- Common project structures
- Repeated decision criteria

Spark may use this learning to offer better defaults.

---

## Anti-Rules

Do not lock members into a rigid template.

Do not silently delete member content.

Do not silently add sections without asking.

Do not make the framework feel like a form.

Do not require every section to be filled before creating the map.

---

## Success Criteria

A good framework gives the member enough structure to begin without making them feel boxed in.
