## Mission

Spark Estate shall continuously become more capable while simultaneously becoming easier to use.

Every release must reduce cognitive effort.

The platform exists to remove mental load from the user — not transfer it to them.

---

## The Missing Piece

As Spark grows more capable — Work, Blueprints, Collections, Chamber, navigation, intelligence — capability alone is not enough.

Without explicit cognitive-load governance, every new power risks becoming another thing the member must learn, choose, configure, or remember.

**128 closes that gap:** capability may grow; member mental load must shrink. This Constitution is the binding check that keeps those two trajectories from diverging.

---

## Constitutional Principle

Users should think about:

- their business
- their ideas
- their goals
- their clients
- their future

Users should never think about:

- the platform
- navigation systems
- configuration
- architecture
- internal concepts

---

## Simplicity Rules (1–15)

### 1. Hide Complexity

If Spark can decide, Spark should decide.

Do not ask the user to make decisions the system can responsibly make.

### 2. Every Screen Answers One Question

Every screen must have:

- one purpose
- one primary action
- one clear next step

### 3. Remove Decisions

The goal is not to give users more options.

The goal is to eliminate unnecessary decisions.

### 4. Progressive Disclosure

Show only what is needed now.

Reveal advanced options only when relevant.

### 5. Never Expose Architecture

Users must never see or need to understand:

- Blueprint Framework
- Registry systems
- Work ID
- Runtime systems
- Identity systems
- Relationship Graph
- Launch Contract
- or any internal architecture

These exist to serve the experience — not to be understood by the user.

### 6. One Thing At A Time

Never ask the user to:

- choose
- configure
- and create

in the same moment.

### 7. Never Punish Exploration

Users must be able to:

- change their mind
- undo
- convert
- move

without losing work.

### 8. Always Protect Momentum

The system must preserve:

- progress
- context
- continuity

Users should never feel like they are starting over.

### 9. Recommend Before Asking

Spark should guide first.

Ask only when necessary.

### 10. Everything Should Feel Recoverable

No action should feel dangerous or irreversible without clear recovery.

### 11. Navigation Should Disappear

Users should not need to learn where things live.

Spark should take them there.

### 12. Every Feature Must Reduce Anxiety

If a feature increases uncertainty, it fails.

### 13. Every Feature Must Reduce Executive Function Load

Spark must reduce the need for:

- memory
- planning
- decision-making
- organization
- task switching
- initiation

### 14. Companion Before Interface

Conversation and guidance come before menus and controls.

### 15. The Ultimate Test

If a user says:

> “It just knows what I need.”

The system passes.

If a user says:

> “I had to figure out the platform.”

The system fails.

---

## Mandatory Simplicity Audit (Before Any Release)

Every feature must answer:

1. What decision did we remove?
2. What complexity did we hide?
3. What cognitive load did we reduce?
4. What anxiety did we eliminate?
5. Can a first-time user succeed immediately?
6. Does this require learning the system?
7. Does this interrupt momentum?
8. Does this expose architecture?
9. Does this create a dead end?
10. Does this feel like Spark — calm, guided, intelligent?

If any answer fails → the feature is not ready.

**Runtime IDs:** `SparkSimplicityAuditQuestionId` in `lib/sparkSimplicityConstitution/types.ts`.

---

## Certification Requirement

No feature may be marked complete (including a perfect functional score) without:

- Functional certification
- Accessibility certification
- Performance certification
- Conversation certification
- **Simplicity certification**
- **Cognitive Load certification**
- **ADHD Experience certification**

A feature that works but overwhelms is incomplete.

**Runtime IDs:** `SparkReleaseCertificationGateId` in `lib/sparkSimplicityConstitution/types.ts`.

---

## Final Constitutional Principle

Spark Estate intelligence is not measured by how much the system can do.

It is measured by how little the user must think about the system while getting meaningful work done.

---

## Order of authority

When this Constitution conflicts with feature convenience, UI novelty, or internal architecture exposure:

1. User trust and safety  
2. **113 Product Constitution**  
3. **This Constitution (128)** on simplicity, cognitive load, and ADHD experience gates  
4. Specialized Constitutions **114–117**  
5. Design Bible (**125**) · Design System (**118–124**)  
6. Feature specifications and implementation docs  

On Work identity and platform source-of-truth, **315 + UWE (112)** still prevail for runtime architecture — but members must never see that architecture (Rule 5).
