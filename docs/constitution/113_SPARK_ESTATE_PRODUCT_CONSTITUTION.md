# 113 — Spark Estate™ Product Constitution
## Master Constitutional Principles

## Purpose

This Constitution defines the non-negotiable product principles for Spark Estate.

It governs:

- Shari
- conversations
- Work
- Projects
- Blueprints
- rooms and navigation
- Chamber members
- the Round Table Board
- Business Pulse
- research
- Cartography
- Body Doubling
- recognition
- memory
- relationships
- accessibility
- future platform experiences

Documents 114–117 expand this Constitution:

- 114 — Shari Companion Constitution
- 115 — Universal Conversation Architecture
- 116 — Estate Navigation Constitution
- 117 — Trust, Memory & Relationship Constitution

The Design System and Design Bible must implement these principles rather than override them.

---

# Article I — The Platform Exists to Help People Move Forward

Spark Estate exists to help people build meaningful businesses without becoming overwhelmed.

Every feature must help the user:

- understand
- think
- decide
- create
- execute
- learn
- recognize progress
- continue

A feature that adds complexity without meaningful forward movement should not be approved.

---

# Article II — Spark Estate Is a Companion Platform

Spark Estate is not primarily:

- a dashboard
- a collection of tools
- a document editor
- a project manager
- a library of disconnected AI assistants

Spark Estate is a connected business companion.

Shari is the primary human-feeling expression of that companionship.

The platform should feel:

- calm
- intelligent
- trustworthy
- connected
- encouraging
- capable
- human

---

# Article III — Conversation Comes First

Conversation is the preferred way to begin, continue, and navigate.

Menus support discovery.

Rooms support focus.

Structured interfaces support completion.

The user should not be required to know where a capability lives before asking for help.

Conversation must remain natural and context-aware rather than robotic, scripted, or menu-like.

---

# Article IV — Meaningful Work May Begin Anywhere

A user may begin meaningful work in:

- Create
- Projects
- Strategies
- Blueprints
- Cartography
- Body Doubling
- Shari conversation
- Chamber
- Board
- Research
- Clear My Mind
- Tasks
- Welcome Home
- any future appropriate experience

The origin must not reduce the quality of the experience.

The platform must bring the correct capability to the user.

---

# Article V — One Work Identity

Meaningful Work must preserve:

- one canonical identity
- one source of truth
- one history
- one current focus
- one relationship graph
- one access model
- one recovery path

Different experiences may view or contribute to the same Work.

They may not create competing copies.

Relationships are preferred over duplication.

---

# Article VI — The User Does Not Manage the Architecture

Users should not need to understand:

- which module owns saving
- which room owns the document
- where a registry lives
- which database table stores the record
- whether a Blueprint or Project is technically primary

The platform must absorb that complexity.

Technical boundaries should protect the user, not become the user's burden.

---

# Article VII — Calm Is a Functional Requirement

Calm is not decoration.

Calm supports:

- comprehension
- trust
- decision-making
- persistence
- accessibility
- ADHD needs
- recovery from overwhelm

The platform must prefer:

- progressive disclosure
- one primary action
- clear language
- visible reassurance
- stable layouts
- calm transitions
- optional depth

A dense interface that exposes everything at once violates this Constitution.

---

# Article VIII — Accessibility Is Foundational

Accessibility is not a final review step.

Every experience must support:

- readable typography
- keyboard use
- screen readers
- visible focus
- contrast
- reduced motion
- sensory-safe sound
- plain language
- error recovery
- predictable interaction

ADHD-friendly design is part of accessibility.

The platform must reduce working-memory burden and decision fatigue.

---

# Article IX — Preserve Context and Return Location

The platform must remember:

- what the user was working on
- why it mattered
- where they stopped
- the current section
- the current question
- the active decision
- related people and Work
- the next helpful step

Users should be able to leave and return without reconstructing context.

A forced restart is a product failure.

---

# Article X — Guidance Must Preserve Agency

Spark Estate may:

- recommend
- explain
- compare
- surface risks
- ask thoughtful questions
- suggest next steps
- provide expert guidance

Spark Estate must not:

- coerce
- shame
- prescribe without context
- silently change Work
- apply advice without approval
- hide uncertainty
- overwhelm the user with options

The user remains the decision-maker.

---

# Article XI — Shari Must Feel Human

Shari should sound:

- warm
- natural
- clear
- confident
- kind
- context-aware
- nonjudgmental

Shari should not sound:

- robotic
- repetitive
- overly cheerful
- clinical
- scripted
- menu-driven
- patronizing

Shari should help the user think before routing them elsewhere unless the user explicitly asks for more resources or direct action.

---

# Article XII — Blueprints Are Reusable Systems

Blueprints are not merely templates.

They are reusable business systems that can include:

- sections
- questions
- defaults
- tasks
- milestones
- research
- risks
- deliverables
- quality checks
- guidance
- relationships

Blueprints support:

## Guided Mode
Complete the work.

## Builder Mode
Improve the reusable system.

## Blueprint Home
Understand, manage, version, and reuse the system.

Blueprints must remain optional and adaptable.

---

# Article XIII — Projects Execute; Work Preserves Meaning

Projects manage execution.

They may contain:

- tasks
- subtasks
- owners
- dates
- dependencies
- milestones
- status

Structured Work preserves substantive meaning, decisions, and deliverables.

Projects and Work should connect through relationships and approved synchronization.

Neither should become a duplicate of the other.

---

# Article XIV — Intelligence Must Be Integrated

Chamber members, the Round Table Board, Research, Business Pulse, and future intelligence experiences must contribute to the connected platform.

They may not become isolated chatbots or private document systems.

Their contributions must:

- preserve attribution
- attach to the correct context
- remain reviewable
- require approval before changing Work
- use shared platform relationships
- preserve trust

---

# Article XV — Research Must Inform, Not Overwrite

Research should improve decisions.

It must preserve:

- sources
- findings
- dates
- confidence
- relevance
- affected decisions
- proposed actions

Research follows:

Research  
→ Review  
→ Approve  
→ Apply

Research must not silently rewrite Work.

---

# Article XVI — Recognition Must Be Meaningful

The platform should recognize meaningful progress without turning the experience into gamification.

Wins, Accomplishments, and Evidence each serve different purposes.

Recognition should:

- reinforce momentum
- make progress visible
- preserve meaningful evidence
- celebrate without pressure
- remain optional
- respect sensory and personal preferences

---

# Article XVII — Memory Must Serve the User

Memory exists to reduce friction and improve continuity.

The platform may remember:

- preferences
- working patterns
- helpful supports
- business context
- relationships
- prior decisions
- unfinished Work

Memory must not:

- surprise the user
- expose confidential information
- create false certainty
- override current instructions
- become manipulative
- retain information beyond appropriate scope

The user must remain able to correct, decline, or update remembered information.

---

# Article XVIII — Relationships Are Preferred Over Copies

When two things are connected, the platform should create a relationship rather than duplicate content.

Examples:

- Work connected to Project
- Work connected to Blueprint
- Work connected to Goal
- Research connected to Decision
- Task connected to Section
- Cartography node connected to Work
- Win connected to Progress
- Accomplishment connected to completed Work

Copies create drift.

Relationships preserve coherence.

---

# Article XIX — Features Must Be Safely Ignorable

Users should be able to ignore a feature without breaking their experience.

The platform may invite.

It should not pressure.

Optional rooms, guidance, recognition, sounds, research, Board reviews, and Chamber contributions must remain optional unless required for safety, integrity, or an explicitly chosen workflow.

---

# Article XX — One Obvious Primary Action

Each screen, room, or moment should present one obvious primary action.

Secondary actions may remain available through progressive disclosure.

When everything looks equally important, the interface has failed to guide the user.

---

# Article XXI — Honest Failure Is Better Than False Success

The platform must never falsely claim:

- work was saved
- a task was created
- research was completed
- a connection exists
- a reminder was scheduled
- a change was applied

When something fails, Spark Estate should:

- say what happened
- preserve what it can
- provide a safe next step
- avoid blame
- retain recovery context

Trust is more important than appearing seamless.

---

# Article XXII — Certification Is Required

A feature is not complete because it renders.

It must prove:

- correct architecture
- correct ownership
- data integrity
- real browser behavior
- accessibility
- ADHD usability
- privacy and security
- domain quality
- recovery
- regression safety
- observability

Production certification must be evidence-based.

---

# Article XXIII — Future Features Must Fit the Whole Estate

Every future feature must answer:

1. How does it reduce cognitive load?
2. How does it preserve Work identity?
3. How does it preserve return location?
4. How does it connect through relationships?
5. How does Shari support it?
6. How does it fit Business Pulse?
7. How does it support accessibility and ADHD needs?
8. What is its one primary action?
9. Can the user ignore it safely?
10. How will it be certified?

A feature that cannot answer these questions should not be built yet.

---

# Article — Capture Before Classification™

**Platform law (Prompt 137).** Extends Article IV and Spec **128** / **132**.

Spark Estate must never require classification, filing, sectioning, labeling, or destination choice before a member can capture.

This applies across:

- Projects (tasks land in Inbox; sections are optional and later)
- Tasks
- Notes and ideas
- Evidence
- Journal
- Clear My Mind (already capture-first — the reference pattern)
- Parking Lot / Park It
- any future capture surface

**Capture first. Organize later. Auto-organize whenever possible.**

Members may group when ready. Spark may gently suggest grouping. Spark must never block capture behind “Where should this go?”

Cursor rule: `.cursor/rules/capture-before-classification.mdc`  
Certification track: `docs/platform/137_*`

# Constitutional Review Test

Before approving any implementation, ask:

- Does this help the user move forward?
- Does it simplify rather than complicate?
- Does it preserve trust?
- Does it preserve one Work identity?
- Does it protect continuity?
- Does it use relationships rather than copies?
- Does it sound and feel like Spark Estate?
- Does it remain accessible?
- Does it preserve user agency?
- Does it avoid unnecessary overwhelm?

If not, redesign.

---

# Order of Authority

When documents conflict, use this order:

1. User trust and safety
2. This Product Constitution
3. The specialized Constitutions in 114–117
4. The Design Bible
5. Certified Product Specifications
6. Design System and UX Standards
7. Implementation documentation
8. Local feature convenience

No lower-level implementation may override constitutional principles for convenience.

# Success Definition

Spark Estate succeeds when users consistently feel:

- "I know what I am working on."
- "I know why it matters."
- "I can begin wherever I am."
- "I can continue where I left off."
- "I do not feel overwhelmed."
- "My work is connected."
- "The platform remembers what helps me."
- "I remain in control."
- "I trust Shari."
- "I trust Spark Estate."

---

# Governance

This Constitution is the master constitutional reference for Spark Estate.

Documents 114–117 expand specific areas but must remain consistent with it.

Any implementation that breaks trust, continuity, Work identity, accessibility, calm, relationships, or user agency is unconstitutional and must be revised.
