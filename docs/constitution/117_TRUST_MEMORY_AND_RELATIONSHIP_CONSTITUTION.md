
# 117 — Trust, Memory & Relationship Constitution

Purpose:
Define how Spark Estate earns and preserves trust.

Trust Rules:
- Never lose user work.
- Never duplicate Work identities.
- Explain important automated actions.
- Keep relationships instead of creating copies.
- Remember only what benefits the user.
- Allow review and correction.
- Preserve privacy and user control.

Relationship Philosophy:
Work is connected to Projects, Goals, Blueprints, Calendar, Visual Thinking, Wins, Accomplishments and Evidence through relationships—not duplicated data.

---

## Relationship Integrity Rule™ (Prompt 141)

**One business memory, many windows.** Everything Knows What It Belongs To — without inventing belonging.

Durable relationship edges may be created **only** from:

1. **Creation flow lineage** — Work created inside a Project, Blueprint, Map, or with an explicit related Work id  
2. **Explicit member link** — Connect / Related To chosen by the member  
3. **Shared Work / Project context** — already belonging to the same container  
4. **Member confirmation** — when a candidate match is ambiguous  

**Never** invent relationships from similar titles or names alone. Fuzzy title overlap may suggest duplicate risk; it must not write graph edges without confirmation.
