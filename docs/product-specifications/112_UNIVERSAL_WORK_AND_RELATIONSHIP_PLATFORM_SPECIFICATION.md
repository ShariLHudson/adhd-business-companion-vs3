# 112 — Spark Estate™ Universal Work & Relationship Platform Specification

## Purpose

This document defines how meaningful work, reusable Blueprints, Projects, conversations, research, visual thinking, guidance, execution, and recognition operate as one connected platform.

It completes the Spark Estate Product Specification Series by establishing the product model that connects:

- the Universal Work Engine
- Create experiences
- Projects
- Blueprints
- Strategies
- Cartography
- Body Doubling
- Shari
- Chamber members
- the Round Table Board
- Research
- Tasks and milestones
- Business Pulse
- Evidence, Wins, and Accomplishments

The user may begin anywhere. Spark Estate must preserve one coherent model of work.

---

# 1. Product Promise

Spark Estate should never require users to understand which feature, room, or technical module owns their work.

A person may begin with:

- a conversation
- an idea
- a Project
- a Strategy
- a Blueprint
- a Cartography node
- a Body Doubling session
- a Chamber conversation
- a Board review
- research
- a task
- Clear My Mind
- Welcome Home
- Create

Where the work begins must not determine the quality, continuity, or durability of the experience.

The platform brings the appropriate work capability to the user.

---

# 2. Governing Principle

> Meaningful work may begin anywhere in Spark Estate, but it must resolve into one connected source of truth with one canonical Work identity, one history, and approved relationships.

This means:

- no shadow workspaces
- no destination-specific duplicate documents
- no private save paths
- no competing registries
- no copying full work content between rooms
- no loss of context when changing experiences
- no forced restart because the user began in the "wrong" place

---

# 3. One Work, Many Views

A Work item is the durable source of truth.

The same Work may be viewed or supported through:

- Create
- Project execution
- Blueprint guidance
- Strategy thinking
- Cartography visualization
- Body Doubling focus
- Shari conversation
- Chamber expertise
- Board review
- Research
- Tasks
- Calendar
- Business Pulse
- Evidence and recognition

These are views and contributions around the same Work.

They are not independent copies.

---

# 4. Canonical Work Identity

Every meaningful master Work item must have:

- one canonical Work ID
- one Work Type
- one schema version
- one lifecycle state
- one current version
- one history
- one current focus
- one relationship graph
- one access model
- one archive and recovery path

Legacy identities may be adopted or retained as aliases, but they may not remain competing master identities.

The canonical Work ID must survive:

- save
- autosave
- refresh
- resume
- reopen from another origin
- Project linking
- Blueprint use
- research
- Chamber contribution
- Board review
- Cartography linking
- Body Doubling
- print or export
- archive and restore

---

# 5. Universal Work Engine

The Universal Work Engine owns platform-wide work behavior.

It owns:

- Work identity
- Work Type registration
- schema initialization
- lifecycle
- section state
- current focus
- save and autosave
- recovery
- exact resume
- versions
- commands
- outputs
- relationships
- Project bridge
- tasks and milestones
- research attachment
- Chamber contributions
- Board reviews
- Shari work context
- Cartography relationships
- archive, Trash, restore, and delete
- universal certification

A Work Type may provide domain intelligence and configuration.

A Work Type may not own a competing runtime.

---

Each Work Type package may define:

- domain schema
- sections
- adaptive questions
- terminology
- Blueprints
- defaults
- commonly forgotten items
- risks
- deliverables
- research prompts
- task recommendations
- milestone recommendations
- Chamber routing preferences
- Board review recommendations
- domain certification tests

A Work Type package may not independently own:

- Work IDs
- persistence
- registry
- commands
- versions
- outputs
- resume
- lifecycle
- relationship storage
- task infrastructure
- research infrastructure

---

# 7. Blueprint Platform

Blueprints are reusable business systems.

A Blueprint may provide:

- purpose
- compatible Work Types
- depth modes
- sections
- adaptive questions
- defaults
- task suggestions
- milestones
- research prompts
- risk prompts
- deliverables
- quality checks
- completion guidance
- Chamber and Board recommendations

Blueprint categories include:

- Spark Blueprints
- Personal Blueprints
- Company Blueprints
- Adaptive Blueprints
- Previous-Work-Derived Blueprints

Blueprints initialize or guide Work.

They do not become a second copy of the Work.

---

# 8. Blueprint Experience Modes

## Guided Mode

Guided Mode helps the user complete the work.

It should:

- ask relevant questions
- explain why something matters
- use examples
- reuse approved known information
- provide a clear next step
- preserve progress
- help the user return later

## Builder Mode

Builder Mode helps the user improve the reusable system.

It should support:

- changing section structure
- improving questions
- changing defaults
- adding forgotten-item intelligence
- refining task and milestone recommendations
- improving quality checks
- creating a new Blueprint version

Builder Mode changes the Blueprint system, not merely one Work instance.

## Blueprint Home

Blueprint Home helps the user understand and manage the Blueprint.

It may include:

- purpose
- compatible Work Types
- versions
- usage history
- Personal or Company scope
- current status
- related Work
- recent improvements
- review needs
- archive and restore
- one clear primary action

It must not become a dense dashboard.

---

# 9. Start Paths

Every compatible creation experience must support:

- Start From Scratch
- Start From Blueprint
- Build From Previous Work

The platform should recommend a path when context is strong, but the user remains in control.

Starting from scratch must remain valid.

Blueprint use must remain optional.

Previous work may be reused selectively with visible provenance.

---

# 10. Anywhere-Origin Resolution

Before creating new Work, Spark Estate should determine:

1. Is the user only talking or reflecting?
2. Does related Work already exist?
3. Should this connect to existing Work?
4. Does this deserve a new master Work item?
5. Is there a compatible Blueprint?
6. What known context may be reused?
7. What is the smallest helpful next step?

The engine resolves one of four outcomes:

- Continue existing Work
- Connect existing Work
- Create new Work
- Remain conversation, research, or support only

The platform should ask only when ambiguity materially changes the result.

---

# 11. Project Relationship

Create owns the substantive Work content.

Projects manages execution.

A Project may:

- execute one Work item
- connect several Work items
- remain a simple master item when deeper structured Work is unnecessary
- contain tasks, subtasks, dates, owners, dependencies, milestones, and status

A Project must not duplicate the full Work content.

Changes between Work and Project execution require approved synchronization.

---

# 12. Research Relationship

Research may attach to:

- Work
- section
- task
- milestone
- decision
- Project
- Blueprint
- Cartography node

Research follows:

Research  
→ Review  
→ Approve  
→ Apply  
→ Version and history update

Research must never silently overwrite Work.

Research records should preserve:

- question
- mode
- sources
- findings
- date
- confidence
- relevance
- affected decisions
- proposed actions
- approval state
- applied changes
- originating experience

---

# 13. Cartography Relationship

Cartography visualizes relationships around Work.

It may show:

- supports
- depends on
- contains
- implements
- informs
- related to
- reused from
- part of

Cartography references canonical IDs.

It must not store a duplicate copy of substantive Work content.

---

# 14. Shari Relationship

Shari is the preferred conversational interface.

Shari supports two modes:

## Talk Only

- reflect
- think
- explore
- clarify
- no Work mutation

## Work on This

- propose changes
- apply approved changes
- attach context to the canonical Work item
- preserve version and history

The user should not be forced to navigate before receiving help.

---

# 15. Chamber and Board Relationship

Chamber members contribute domain expertise.

The Round Table Board contributes review, perspective, tradeoffs, and decision support.

Contributions must:

- retain attribution
- preserve source context
- attach to the correct Work, section, or decision
- remain advisory until approved
- never create private duplicate workspaces
- use the Universal Work Engine

---

# 16. Body Doubling Relationship

Body Doubling may attach to:

- Work
- section
- task
- milestone

The session should preserve:

- focus
- session purpose
- progress
- notes
- next step
- relationship to the canonical Work ID

Body Doubling must not create a competing private checklist when structured Work already exists.

---

# 17. Business Pulse Relationship

Business Pulse observes meaningful movement across the connected platform.

It may recognize:

- Work advanced
- a decision was made
- a milestone was completed
- a blocker appeared
- research changed direction
- a Project needs attention
- a meaningful win occurred
- an accomplishment deserves preservation

Business Pulse should quietly answer:

- What moved forward?
- What deserves attention?
- What deserves recognition?
- What is the next helpful step?

It must not become a widget-heavy dashboard.

---

# 18. Recognition Relationship

Recognition must remain purposeful.

## Wins

Meaningful momentum worth noticing.

## Hall of Accomplishments

Durable completed achievements.

## Evidence Vault

Discoveries, patterns, lessons, problems solved, and contributors.

Work completion may contribute to recognition, but the categories must not collapse into one generic activity log.

---

# 19. ADHD and Accessibility Requirements

The connected Work experience must:

- reduce choices
- use progressive disclosure
- show one primary next action
- preserve exact return location
- provide visible save reassurance
- allow skip and return later
- allow "I don't know"
- support low-energy continuation
- avoid forced restart
- avoid duplicate work
- avoid requiring memory of navigation
- use readable typography
- support keyboard and screen readers
- respect reduced motion and sensory preferences

---

# 20. Trust and Safety Requirements

The platform must:

- preserve ownership and access scope
- isolate Personal Blueprints
- enforce Company Blueprint scope
- protect confidential information
- require approval before applying research or advisory contributions
- preserve history
- provide archive and recovery
- fail visibly rather than silently falling through
- avoid claiming work was saved when it was not
- avoid destructive migration without recovery

---

# 21. Certification Standard

A Work capability is not complete until it proves:

- architectural ownership
- automated tests
- real browser behavior
- ADHD usability
- accessibility
- domain quality
- data integrity
- privacy and security
- observability
- production recovery

Release blockers include:

- duplicate master Work
- lost content
- wrong Work reopened
- shadow workspaces
- direct unauthorized persistence
- silent research overwrite
- unapproved Chamber or Board application
- Shari Talk Only mutating Work
- depth changes creating a new Work ID
- origin-specific bypass of the Universal Work Engine

---

# 22. Success Definition

The platform succeeds when users can say:

- "I can start wherever I am."
- "The platform understands what I am working on."
- "I do not have to remember where it lives."
- "I can continue exactly where I left off."
- "My work is connected without being duplicated."
- "I know what the next helpful step is."
- "I feel supported rather than managed."
- "I trust Shari and the platform with my work."

---

# Governance

This specification governs the connected product behavior linking the Product Specification Series to the Spark Estate Constitution.

If a feature creates a private source of truth, duplicates Work, breaks continuity, increases cognitive load, or forces the user to understand technical ownership, it does not meet this specification.
