# 319 — Spark Estate Platform Constitution

## Constitutional status

This document is the supreme governing architecture standard for Spark Estate.

Every Blueprint, Chamber member, Board member, Business Estate feature, Create experience, Project workflow, intelligence file, registry, router, capability, integration, data model, interface, and certification system must comply with this Constitution.

When any other document conflicts with this Constitution, this Constitution governs unless Shari explicitly approves a new constitutional amendment.

## Foundational mission

Spark Estate exists to help people think more clearly, make better decisions, create meaningful work, move forward with less friction, and build lives and businesses that feel more understandable, connected, and possible.

The platform must feel like one intelligent estate, not a collection of disconnected tools.

## Article I — One coherent platform

Spark Estate is one platform.

It must not become:

- a collection of isolated assistants
- a collection of disconnected rooms
- a collection of duplicate databases
- a collection of competing registries
- a collection of separate task systems
- a collection of separate context stores
- a collection of unrelated prompts
- a collection of features that each forget the user

Every major concept must have:

- one authoritative owner
- one stable identity
- one lifecycle
- one source of truth
- controlled extensions
- explicit relationships
- shared certification

## Article II — Single ownership

Every platform concept has one authoritative owner.

Other systems may:

- extend
- configure
- project
- reference
- contribute
- validate
- migrate into

They may not recreate the same ownership.

The following are constitutional owners:

- Universal Work Engine owns Work identity and lifecycle
- Universal Work Type Registry owns Work Type identity and registration
- Universal Create owns creation runtime
- Create Artifact owns created content
- Projects owns execution
- Task owns task identity and lifecycle
- Relationship Registry owns cross-object relationships
- Business Profile owns business identity
- Business DNA owns reusable business context
- Client Avatar owns audience identity
- Canonical Person and Organization own identity
- Client Account owns the business-client relationship
- canonical Offer, Product, Service, and Package own commercial definitions
- Pricing Model owns pricing logic
- global routing owns route resolution
- Universal Work context envelope owns runtime context
- Universal Work Type Certification owns certification authority

## Article III — No shadow systems

A shadow system is any second implementation that stores, routes, resumes, certifies, or governs the same concept outside the constitutional owner.

Shadow systems are prohibited.

Examples include:

- Blueprint-specific Work IDs
- Blueprint-only Project models
- Chamber-specific Task models
- separate Blueprint routers
- separate Blueprint certification authorities
- separate client records by industry
- separate Business Profile copies
- isolated context stores
- duplicate status systems
- hard-coded catalogs that compete with registries
- copied Create content inside Projects
- duplicated profile fields stored as authoritative session data

## Article IV — Stable identity

Every major object must have a stable identity.

Stable identity must survive:

- renaming
- versioning
- navigation changes
- UI redesign
- taxonomy changes
- route changes
- migration
- archival
- restoration
- handoff between experiences

Display names must never serve as authoritative IDs.

## Article V — Relationships over copies

Spark Estate connects objects through explicit relationships.

It does not create disconnected copies when a relationship is sufficient.

Examples:

- Project links to Create Artifact
- Proposal links to Client Account and Offer
- Task links to Project and source content
- Blueprint session links to Business Profile
- event links to Vendor Account
- person links to roles
- dashboard links to Metric Definitions
- client engagement links to Client Account

The governing principle is:

> One canonical record, many legitimate relationships.

## Article VI — Context continuity

Spark Estate must remember what the user has already provided.

The platform must reuse:

- Business Profile
- Business DNA
- People I Help
- Client Avatars
- Offers
- Products
- Services
- active Projects
- Create artifacts
- previous Blueprint work
- user preferences
- accessibility preferences
- current context
- recent decisions

The platform must not repeatedly ask questions whose answers are already known and still valid.

When information conflicts, the platform must distinguish:

- session-only override
- Work-only override
- Project-only override
- Create-only override
- canonical update

Canonical records must not be overwritten without permission.

## Article VII — Create and Projects

Create and Projects are distinct but connected.

Create is the source of truth for created content.

Projects manage execution through:

- tasks
- subtasks
- dates
- milestones
- dependencies
- owners
- risks
- progress

A Create-to-Project handoff must preserve:

- Create identity
- content source
- version
- context
- provenance
- relationships
- user intent

Projects must not duplicate the created artifact.

## Article VIII — Blueprints

A Blueprint is not a separate runtime.

A Blueprint is:

- a registered Work Type
- a configuration of Universal Work
- an orchestration of Create
- a context-aware pathway
- a set of promised outputs
- a capability composition
- a shared-object dependency map
- a routing configuration
- a certification profile

Every Blueprint must:

- have a stable ID
- be registered
- load known context
- suppress repeated questions
- create real outputs
- save
- resume exactly
- reuse canonical objects
- hand off intentionally to Projects
- declare boundaries
- pass certification

No Blueprint may promise an output that cannot actually be created.

## Article IX — Chamber members

A Chamber member is a platform capability, not an isolated chatbot.

Every Chamber member must:

- operate in Shari’s voice
- use shared context
- contribute to Universal Work
- respect canonical owners
- read approved intelligence
- return structured capability outputs
- preserve provenance
- avoid creating duplicate records
- disclose limits
- coordinate with other members when appropriate

One Chamber member may be accountable for an output while others contribute.

## Article X — Board members

The Board supports thinking, judgment, options, tradeoffs, and decision quality.

The Board must not:

- replace licensed professionals
- issue hidden directives
- create disconnected decision records
- override the user
- prescribe as if certainty exists

Board decisions and recommendations must connect to canonical Decision records when saved.

## Article XI — Shari

Shari is the relational intelligence layer of Spark Estate.

Shari must be:

- warm
- clear
- kind
- natural
- confident
- context-aware
- non-robotic
- non-menu-like
- honest
- supportive without being patronizing

Shari should help the user think, choose, create, and move forward.

She should not:

- overwhelm
- repeat known questions
- force navigation
- auto-launch experiences after merely mentioning them
- pretend unavailable capabilities exist
- overstate certainty
- lose context
- use cold system language
- reduce the user to a diagnosis or type

## Article XII — User agency

The user remains in control.

Spark Estate may:

- recommend
- explain
- compare
- clarify
- suggest
- help decide
- ask permission
- support execution

Spark Estate must not:

- silently overwrite
- force routes
- make hidden commitments
- fabricate completion
- impersonate the user
- submit, publish, spend, or send without authorization
- treat inferred preferences as permanent facts without validation

## Article XIII — Progressive disclosure

Spark Estate must not display the entire estate, catalog, or decision space at once when a smaller path would be more helpful.

Default behavior:

- show the next meaningful step
- offer a few strong choices
- explain why they fit
- allow browsing
- preserve access to the full system
- support low-energy paths
- support advanced paths

Companion First. Dashboard Second. Menu Third.

## Article XIV — Accessibility and cognitive load

Accessibility is architectural, not decorative.

The platform must support:

- large readable text
- clear hierarchy
- simple choices
- keyboard accessibility
- understandable labels
- predictable navigation
- saved progress
- exact resume
- reduced repetition
- low-energy mode
- reduced decision burden
- visible consequences
- recoverable mistakes

No feature is complete if it technically works but creates avoidable cognitive overload.

## Article XV — Truthfulness

Spark Estate must represent reality honestly.

It must distinguish:

- implemented
- partially implemented
- future
- unavailable
- blocked
- inferred
- assumed
- user-provided
- externally verified
- AI-generated

The platform must never imply that:

- an output exists when it does not
- a route works when it does not
- a capability is connected when it is not
- a professional review occurred when it did not
- a migration succeeded without evidence
- certification passed without tests

## Article XVI — Professional and safety boundaries

Spark Estate may support business operations, thinking, planning, documentation, and decision preparation.

It must not present itself as a replacement for:

- legal counsel
- medical care
- financial licensure
- accounting review
- insurance advice
- emergency response
- regulated professional judgment

Professional-review flags must be explicit where needed.

## Article XVII — Data stewardship

Every canonical object must support:

- ownership
- permissions
- business isolation
- provenance
- created date
- updated date
- archive state
- version where needed
- relationships
- retention
- migration
- restoration

No cross-business or cross-client data leakage is acceptable.

## Article XVIII — Registries

Registries are authoritative architecture, not optional documentation.

Required registries include:

- Work Type Registry
- Object Type Registry
- Field Registry
- Relationship Type Registry
- Output Type Registry
- Capability Registry
- Context Dependency Registry
- Route Configuration Registry
- Certification Module Registry
- Extension Registry
- Migration Registry
- Deprecation Registry

User-facing lists must derive from or validate against registries.

## Article XIX — Versioning and migration

All significant architecture must support versioning.

Migrations must:

- preserve user data
- preserve stable IDs
- preserve provenance
- preserve relationships
- preserve permissions
- preserve exact resume
- support validation
- support rollback

No destructive migration is allowed without evidence and approval.

## Article XX — Certification

Certification must prove runtime behavior.

Documentation alone is not certification.

Required certification may include:

- Universal Work Type
- Blueprint Createability
- Blueprint Context Connection
- Shared Object compliance
- No-Duplication
- routing
- exact resume
- accessibility
- privacy
- domain validity
- professional boundaries
- Shari voice
- migration integrity

Certification statuses:

- pass
- pass with declared limits
- fail
- blocked

## Article XXI — Documentation precedence

When documents conflict, apply this order:

1. Spark Estate Platform Constitution
2. Platform Authority Hierarchy
3. Universal Work and Work Type standards
4. Universal Create and Create Artifact standards
5. Projects and Task standards
6. Relationship Registry standards
7. canonical Business Profile and Business DNA standards
8. shared-object standards
9. Blueprint standards
10. Chamber and Board standards
11. industry-specific files
12. templates and examples

Later files do not automatically supersede higher-order architecture.

## Article XXII — Future expansion

New features must first answer:

- What constitutional owner does this extend?
- What object does it use?
- What relationship does it create?
- What context does it require?
- What output does it produce?
- Where does it save?
- How does it resume?
- What certification applies?
- What duplicate-system risk exists?

A new system may be created only when no constitutional owner can legitimately support it.

## Article XXIII — Definition of complete

A feature is complete only when:

- it works
- it uses the correct owner
- it creates no shadow system
- it preserves user context
- it saves
- it resumes
- it is accessible
- it is honest
- it is testable
- it is certified
- it can be maintained
- it fits the estate as one coherent platform

## Constitutional principle

> One estate. One architecture. One source of truth. Many connected experiences.
