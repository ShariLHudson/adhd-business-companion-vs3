# 532 — Knowledge Lifecycle, Versioning & Canonical Truth Standard

## Knowledge lifecycle

1. captured
2. classified
3. reviewed
4. validated
5. approved
6. published
7. used
8. monitored
9. refreshed
10. superseded
11. archived
12. retired

## Canonical truth rule

For each domain, Spark Estate should identify one current authoritative representation.

Other versions may exist as:

- historical
- local adaptation
- project-specific
- client-specific
- draft
- experimental
- superseded

## Version requirements

Each version should include:

- version number
- effective date
- owner
- change summary
- source changes
- approval
- affected relationships
- migration or update notes
- previous version

## Supersession behavior

When new knowledge replaces old knowledge:

- preserve history
- mark the old item clearly
- update retrieval priority
- identify affected outputs
- avoid silent deletion
- preserve links and lineage

## Review triggers

- scheduled review
- new evidence
- policy change
- regulation change
- business change
- repeated user correction
- contradiction
- poor outcome
- new canonical owner
- source expiration
