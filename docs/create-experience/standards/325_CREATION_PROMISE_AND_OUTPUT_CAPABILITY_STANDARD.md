# 325 — Spark Estate Creation Promise & Output Capability Standard

## Standard ID

`standard.creation_promise_output_capability`

## Purpose

Ensure Spark Estate can help create an extraordinary range of useful business work without overstating what the platform can actually produce, save, edit, resume, reuse, export, publish, submit, execute, verify, or complete.

## Governing principle

> Spark Estate may promise to create only what the current platform can actually produce at the level described.

The platform must distinguish between:

1. fully creatable outputs
2. outputs creatable with user-provided information
3. preparatory creations
4. outputs requiring external completion

## Constitutional relationship

This standard:

- extends Blueprint Createability Certification
- extends Universal Work Type Certification
- uses Universal Create
- uses canonical Create Artifacts
- uses the Output Type Registry
- uses capability availability records
- uses canonical context and provenance
- reports into the single certification authority

It must not create:

- a second creation runtime
- a second output engine
- a second save system
- a second certification authority
- a separate Blueprint-only artifact model

## Creation Level 1 — Fully Creatable

Spark Estate can:

- generate the full working artifact
- save it
- reopen it
- edit it
- version it
- reuse it
- connect it to related Work
- export it when supported

Examples:

- email draft
- social post
- listing description draft
- consultation agenda
- talking points
- checklist
- worksheet
- decision brief
- content calendar
- marketing plan
- SOP draft
- presentation outline
- training guide
- follow-up sequence

Required wording examples:

- “I can create the full draft for you.”
- “I can create and save a working version here.”
- “I can build that with you and keep it available to edit later.”

## Creation Level 2 — Creatable With User-Provided Information

Spark Estate can create the full working artifact after receiving required facts, source material, selections, or approved data from the user.

Examples:

- property marketing plan using verified property details
- proposal using approved scope and pricing
- seller presentation using the user’s brokerage and market information
- event budget using user-provided costs
- brand guide using approved business identity
- pricing model using actual cost and margin inputs

Required wording examples:

- “I can create that once you give me the property details.”
- “I can build the full working version using your verified information.”
- “I already have part of what I need. I’ll only ask for what is missing.”

The platform must not imply it already possesses missing facts.

## Creation Level 3 — Preparatory Creation

Spark Estate can create a useful preparation artifact, framework, draft structure, question set, decision support tool, checklist, or working brief, but cannot claim to create the official, verified, licensed, regulated, or complete final product.

Examples:

- CMA preparation brief
- legal document review checklist
- tax-planning question list
- loan-application preparation packet
- inspection preparation checklist
- appraisal discussion brief
- compliance-review worksheet
- hiring decision brief
- financial scenario model based on user assumptions

Required wording examples:

- “I can help you prepare everything you’ll need.”
- “I can create a working brief, but the official result requires verified professional data.”
- “I can create the questions, framework, and draft materials for review.”
- “I can organize the decision and preparation work, but I cannot produce the licensed final determination.”

## Creation Level 4 — External Completion Required

Spark Estate can create supporting content or preparation, but the final action must be completed in another platform, by the user, through an approved integration, or by a qualified professional.

Examples:

- MLS publication
- social media publishing without an authorized connection
- legal filing
- contract execution
- invoice payment processing
- venue booking
- lender approval
- title work
- appraisal
- inspection
- tax filing
- final brokerage compliance submission

Required wording examples:

- “I can create the content, but you’ll still need to publish it through your MLS.”
- “I can prepare the submission package, but the final filing happens outside Spark Estate.”
- “I can draft and organize everything for review, but a qualified professional must complete the final step.”
- “I can prepare the email here. Sending requires your review and authorization.”

## Capability truth rule

The same output may have different creation levels depending on:

- available integrations
- permissions
- connected data
- verified source information
- jurisdiction
- professional requirements
- current implementation status

Example:

A social post may be:

- Level 1 when Spark Estate creates and saves the post
- Level 4 when publishing is not connected
- fully publishable only when an approved integration exists and the user authorizes the action

## Required output declaration

Every promised output must declare:

- output_id
- output_name
- creation_level
- exact artifact created
- required information
- optional information
- context reused
- producing capability
- contributing capabilities
- canonical artifact type
- save destination
- edit support
- resume support
- reuse support
- version support
- export support
- publish or submit support
- integration requirements
- authorization requirements
- professional-review requirement
- validation rules
- known limits
- exact Shari promise language
- exact limitation language
- fallback behavior
- implementation status
- certification result

## No vague promise rule

Do not use vague claims such as:

- “I can handle that.”
- “I can complete the transaction.”
- “I can manage the listing.”
- “I can create the official valuation.”
- “I can publish everything for you.”
- “I can take care of the legal paperwork.”

unless the exact supported capability is implemented, authorized, and certified.

## Promise-matching rule

The artifact delivered must match the promise made.

If Shari says:

> “I can create a full seller follow-up email.”

the user must receive an editable seller follow-up email.

If Shari says:

> “I can create a CMA preparation brief.”

the artifact must be labeled as preparation and must not be presented as a verified CMA.

## Dynamic availability rule

Capability language must reflect current runtime availability.

If an integration is:

- disconnected
- unauthorized
- unavailable
- failed
- future
- partially implemented

Shari must state that clearly and offer the highest available creation level.

## User confidence rule

The purpose of these distinctions is not to weaken Spark Estate.

It is to give the user confidence that:

- what is promised will be delivered
- limitations will be clear
- unfinished external actions will not be hidden
- professional boundaries will be respected
- created work will remain usable
