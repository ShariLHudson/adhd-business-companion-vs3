# 047 — Create Ecosystem and Asset Generation Standard

**Status:** Binding  
**Parents:** [045 Platform Intent Routing](./045_PLATFORM_INTENT_ROUTING_AND_CREATION_STANDARD.md) · [046 Create Blueprint Standard](./046_CREATE_BLUEPRINT_STANDARD.md)  
**Companion:** [048 Creation Workspace](./048_CREATION_WORKSPACE_STANDARD.md) · [049 Connection & Relationship Registry](./049_CREATION_ECOSYSTEM_CONNECTION_STANDARD.md)  
**Runtime:** `lib/createAssets/`

## Mission

Every Create Blueprint represents a complete creation ecosystem—not a single document.

When a user creates something, Spark Estate must understand:

- the primary creation
- every supporting asset that may be needed
- which assets already exist
- which assets should be suggested
- which assets can be generated automatically
- how every asset connects to the same canonical Creation Record

The user should never need to wonder where to make the registration page, speaker packet, or volunteer handbook. Everything belongs to the same creation.

## One Creation = One Ecosystem

A Retreat Weekend is **not** only a Retreat Plan. It is one event with many connected assets (plan, budget, timeline, registration, emails, speaker packets, volunteer handbook, run of show, debrief, archive, …).

One event. Many connected assets. One canonical Creation Record.

## Create Asset Registry (shared platform service)

Blueprints do **not** hard-code private template copies.

Each blueprint **references** reusable assets from the Create Asset Registry.

Example: “Confirmation Email”, “Budget Worksheet”, “Timeline”, and “Checklist” are each one asset. Retreat, Conference, Workshop, and Webinar blueprints all use those definitions and customize them for context.

That keeps the platform consistent, avoids duplicate maintenance, and improves one asset everywhere it is used.

## Asset Intelligence

Every Blueprint owns an **Asset Registry view** (references into the shared registry).

Each asset contains:

| Field | Purpose |
|-------|---------|
| Asset Name | Member-facing label |
| Purpose | Why it exists |
| Template | Editable builder / template id |
| Editable | Can be created inside Spark Estate |
| Required? / Optional? | Planning guidance — never dump all |
| Dependencies | Assets or decisions that unlock this one |
| Generated by | Runtime / Chamber / user |
| Connected Chamber Members | Primary + supporting |
| Connected Board advisors | Advise only — never edit |
| Can become Project Task? | Project integration hint |
| Can appear in Cartography? | Graph node eligibility |
| Completion Rules | When “done enough” |

## Dynamic Asset Suggestions

Spark Estate must never dump dozens of assets on the screen.

After each planning phase:

1. Evaluate what is true now
2. Suggest only the next logical assets (max 3)
3. One-click generate when the user accepts

Example after venue is selected:

> You've selected the venue. The next things most organizers usually create are: Registration Form · Venue Timeline · Vendor Checklist. Would you like me to build one?

## Chamber and Board

- Every asset declares a **primary** Chamber member and optional **supporting** members.
- No duplicated intelligence. One owner. Others contribute.
- Board advisors may be suggested for decisions. The Board never edits. The Board advises.

## Create Must Become the Builder

Users create assets inside Spark Estate—editable, saved, connected—not blank Word downloads.

Builders (agenda, session, registration, email, volunteer schedule, budget, run of show, checklists, timelines, etc.) are the long-term surface. V1 may scaffold records that those builders fill.

## Automatic Project Integration

Every generated asset asks (internally): Should this become work?

| Answer | Behavior |
|--------|----------|
| No | Asset only |
| Reference Only | Linked, no tasks |
| Suggested Tasks | Offer tasks after confirmation |
| Required Tasks | Create confirmed tasks when appropriate |

Project Home updates automatically when beneficial (045 / 046).

## Cartography Integration

Every creation ecosystem can become a visual graph. Nodes open the actual asset. Visible Thinking remains permission-based (046).

## Universal Create Rule

Every Blueprint must answer:

> What else is normally created when someone creates this?

Those assets become **suggestions**. Never assumptions. Never forced dumps.

## Cross-Blueprint Relationships

Assets may belong to multiple Blueprint types (e.g. Email Sequence used by Events, Marketing, Course, Membership, Launch, Client Onboarding).

One asset definition. Multiple Blueprint references. Never duplicate templates.

## Universal User Experience

The user never thinks “I need another document.”

Shari notices and offers the next natural asset. The platform thinks ahead. The user stays focused.

## Architecture Stack

| Layer | Role |
|-------|------|
| **045 Intent Routing** | What the user is trying to accomplish |
| **046 Blueprints** | How that type of creation is built |

This scales from a single proposal to a multi-day conference while feeling like one continuous experience.
