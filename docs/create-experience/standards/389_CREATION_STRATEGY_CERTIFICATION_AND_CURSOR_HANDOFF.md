# 389 — Creation Strategy Certification & Cursor Handoff

## Certification ID

`certification.creation_strategy_intelligence_12_10`

## Architecture gates

- no duplicate Create runtime
- no duplicate output registry
- no duplicate relationship model
- no duplicate visual engine
- no duplicate Project logic
- no duplicate Blueprint logic
- no duplicate knowledge store
- no duplicate certification authority

## Functional gates

### Strategy selection

- create, reuse, extend, adapt, merge, split, connect, or no-action recommendations work
- one recommended action is clear
- simple requests are not delayed

### Duplicate prevention

- near-duplicates are detected
- meaningful variants remain separate
- no silent merging occurs
- aliases use canonical output identity

### Scope

- smallest sufficient artifact is preferred
- asset sets are recommended only when valuable
- overbuilding is prevented
- Blueprint and Project boundaries remain clear

### Knowledge boundary

- knowledge candidates may be prepared
- no candidate is declared canonical prematurely
- provenance is retained
- approval is required

### Lifecycle

- consolidation, split, archival, and retirement are safe
- destructive actions require authorization
- lineage survives replacement or merge

### Value loop

- real-world use feedback is optional and respectful
- outcome evidence improves strategy
- no moral productivity scoring occurs

## Required Cursor deliverables

- current artifact search and similarity audit
- existing duplicate-detection inventory
- canonical owner map
- strategy trigger implementation
- similarity and overlap logic
- scope classifier
- reusable-asset detector
- knowledge-candidate handoff
- safe lifecycle actions
- tests
- files changed
- migrations
- known limits
- certification result

## Final principle

> Before Spark Estate creates more, it should quietly determine whether the user would be better served by creating, reusing, extending, adapting, connecting, or simplifying what already exists.
