# 105 — Marketing Plan Work Type Foundation

## Start condition

Do not begin this implementation unless:

- Universal Work Engine foundation is complete
- Universal Blueprint Framework is complete
- durable Blueprint storage is complete
- Universal Blueprint Interface is complete
- Anywhere-Origin routing is complete
- Event Anywhere-Origin certification verdict is `WORK TYPE PRODUCTION CERTIFIED`
- no release blockers remain

If any start condition is false, stop and report the missing prerequisite. Do not build around it.

## Purpose

Build Marketing Plan as the first non-Event Work Type and the first proof that the Universal Work Engine is genuinely domain-independent.

Start with one Blueprint only:

- Simple Marketing Plan

Do not add the other Marketing Plan Blueprints yet.

Do not copy Event code.

Do not create a Marketing-specific runtime, registry, save path, task owner, research owner, relationship owner, Project bridge, command system, output owner, version owner, or resume owner.

## Governing quality standard

Marketing Plan must contain expert-level, proven marketing reasoning and implementation guidance in Shari's warm, natural voice.

Do not mention specific expert names.

It must provide more depth and practical intelligence than a normal generic chat response while remaining accessible and not overwhelming.

It must help the user understand, decide, build, implement, and improve a marketing plan.

## Simple Marketing Plan Blueprint

Register one Spark Blueprint through the universal Blueprint registry.

Recommended stable ID:

`marketing_plan.simple`

Use repository conventions if another exact format is already authoritative.

Support:

- Quick Start
- Guided Build
- Complete Planning

All depth modes must preserve the same canonical Work ID.

## Marketing Plan section architecture

Build a practical universal section model for a Simple Marketing Plan.

At minimum include:

1. Purpose and desired outcome
2. Business and offer context
3. People the user wants to reach
4. Current situation
5. Positioning and core message
6. Marketing goals
7. Channels and places to show up
8. Content and communication approach
9. Offer path and calls to action
10. Simple campaign or activity plan
11. Budget, time, energy, and capacity
12. Measures and signals
13. Risks, assumptions, and what may be missing
14. Next actions
15. Review and improvement rhythm
16. Final plan and deliverables

The exact user-facing names should sound natural and supportive rather than academic or technical.

Do not force all sections to appear in Quick Start.

Use required, conditional, optional, and hidden sections through the Blueprint framework.

## Quick Start requirements

Quick Start should produce a useful first plan with minimal friction.

Ask only what is needed to establish:

- what is being marketed
- who it is for
- the main outcome
- the most realistic channel or approach
- the next few actions
- a simple way to tell whether it is helping

Reuse known business, offer, audience, and brand information with approval.

## Guided Build requirements

Guided Build should add:

- explanations
- examples
- adaptive questions
- likely tradeoffs
- recommended next steps
- channel guidance
- simple messaging development
- activity sequencing
- implementation support

## Complete Planning requirements

Complete Planning should add:

- fuller audience and context analysis
- positioning depth
- channel-role decisions
- funnel or journey thinking where appropriate
- budget and capacity tradeoffs
- risks and assumptions
- testing plan
- measurement plan
- dependencies
- implementation tasks and milestones
- review cadence
- quality checks
- research prompts
- Chamber and Board review recommendations

Do not turn Complete Planning into a rigid corporate marketing document for users who do not need one.

## Adaptive question architecture

Every question must define:

- purpose
- why it matters
- examples
- required/conditional/optional status
- what known context may answer it
- safe inference rules
- postpone behavior
- skip behavior
- "I don't know" behavior
- follow-up rules
- sections affected
- deliverables affected
- tasks affected
- research opportunities
- Quick Start wording
- Guided Build wording
- Complete Planning wording
- low-energy wording

Before asking:

- Is this already known?
- Is it needed now?
- Can it be inferred safely?
- Can it wait?
- Will it materially change the next step?
- Can a lower-friction question work?

## Domain intelligence

Include proven marketing reasoning such as:

- goals must connect to business outcomes
- audience understanding should be specific enough to guide choices
- one clear primary audience is often better than trying to speak to everyone
- positioning should clarify relevance and difference
- channels should have distinct roles
- activity should match capacity
- consistency must be realistic
- measurement should connect to decisions
- plans should include testing and learning
- implementation should be sequenced
- risks and assumptions should be visible
- the plan should remain adaptable

Do not reference named experts.

Do not include unsupported guarantees.

## ADHD-supportive adaptations

Marketing Plan must:

- avoid demanding every answer up front
- offer one helpful next step
- support low-energy mode
- distinguish ideas from commitments
- reduce channel overload
- prevent overbuilding
- surface the smallest useful plan
- show progress
- preserve Parking Lot ideas without forcing them into the active plan
- allow "I don't know"
- allow skip and return
- help convert decisions into visible next actions
- support Body Doubling for section or task work
- avoid shame-based language

## Shari behavior

Support:

### Talk only
Shari helps the user think without changing the plan.

### Work on this
Shari proposes or makes approved changes to the active Work item.

Shari should sound warm, natural, context-aware, and human.

Avoid robotic menus and repeated scripts.

## Body Doubling

Support Body Doubling attached to:

- whole Marketing Plan
- active section
- task
- milestone

A session should be able to resume the exact focus and record progress without creating a separate competing checklist.

## Completion model

Completion should be section-aware and depth-aware.

The Work item may be:

- started
- in progress
- ready for review
- implementation ready
- active
- under review
- updated
- complete
- archived

Do not equate "all questions answered" with "good plan."

Quality checks should consider coherence, feasibility, clarity, implementation readiness, and measurement.
