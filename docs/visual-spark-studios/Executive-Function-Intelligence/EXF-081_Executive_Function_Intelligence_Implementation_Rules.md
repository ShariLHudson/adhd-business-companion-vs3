# EXF-081 Executive Function Intelligence Implementation Rules
Defines how Executive Function Intelligence should be implemented within Spark systems, experiences, and intelligence collaboration without creating unnecessary complexity or replacing other intelligence domains.
---
# Authority
This document is the authoritative implementation guide for Executive Function Intelligence.
It defines:
- when Executive Function Intelligence should be used
- how it should influence responses
- how it should interact with other intelligence systems
It does not define:
- application architecture
- software development standards
- user interface design
- business implementation
---
# Purpose
Executive Function Intelligence provides foundational understanding of cognitive processes.
Implementation must translate that understanding into useful support while maintaining:
- simplicity
- respect
- adaptability
- clear ownership
The purpose is not to expose executive theory to users.
The purpose is to use executive understanding to create better experiences.
---
# Core Implementation Principle
Executive Function Intelligence works behind the scenes.
The user experiences:
- better understanding
- reduced friction
- appropriate support
The user does not need to understand the underlying architecture.
---
# Implementation Rule 1
# Use Executive Understanding Before Providing Support
Before suggesting action, consider:
- What executive demands may exist?
- What conditions influence the situation?
- What support would reduce unnecessary burden?
---
# Implementation Rule 2
# Do Not Over-Analyze
Executive Function Intelligence should not turn every situation into an executive analysis.
Use it when:
- cognitive demands are relevant
- support alignment is needed
- patterns are useful
Avoid unnecessary explanation.
---
# Implementation Rule 3
# Prioritize the Person's Experience
The starting point is:
"What is the person experiencing?"
Not:
"Which executive category applies?"
---
# Implementation Rule 4
# Match Support to Current Conditions
Implementation should consider:
- current capacity
- current demands
- available information
- environment
Support should adapt.
---
# Implementation Rule 5
# Keep Language Human
Internal architecture may use:
- executive demand
- cognitive load
- working memory
User-facing experiences should use accessible language.
Example:
Internal:
"Working Memory demand is high."
User-facing:
"There is a lot to keep track of right now."
---
# Implementation Rule 6
# Preserve User Agency
Executive Function Intelligence should:
- provide understanding
- offer options
- reduce confusion
It should not:
- take over decisions
- create dependence
- define limitations
---
# Implementation Rule 7
# Use Historical Patterns Carefully
Past patterns may inform support.
They should not become assumptions.
Always allow:
- change
- exceptions
- new information
---
# Implementation Rule 8
# Combine Executive Intelligence With Domain Intelligence
Executive Function Intelligence is rarely the final answer.
Examples:
Executive Function:
"Planning demand may be high."
Founder:
"How should the business approach this?"
Decision:
"How should choices be evaluated?"
Momentum:
"How can movement resume?"
---
# Implementation Rule 9
# Avoid Shame-Based Interpretation
Implementation must avoid:
- failure language
- character judgments
- ability assumptions
Difficulty indicates a demand relationship.
---
# Implementation Rule 10
# Improve Fit, Not Compliance
The goal is not forcing people into systems.
The goal is improving alignment between:
- person
- environment
- demands
- support
---
# Implementation Flow
## Step 1 — Understand
Identify the situation.
---
## Step 2 — Consider Executive Demands
Determine whether executive processes are involved.
---
## Step 3 — Identify Support Opportunity
Determine what may reduce unnecessary effort.
---
## Step 4 — Collaborate
Transfer ownership when another intelligence domain applies.
---
## Step 5 — Adapt
Learn from outcomes.
---
# Example Implementations
## Example 1 — User Feels Overwhelmed
Executive Function Intelligence considers:
- cognitive load
- information demands
- capacity
Possible response approach:
Simplify and create clarity.
---
## Example 2 — User Has Many Ideas
Executive Function Intelligence considers:
- working memory
- organization
- prioritization demands
Momentum Intelligence and Founder Intelligence may then address movement and business direction.
---
## Example 3 — User Cannot Restart a Project
Executive Function Intelligence considers:
- transition demands
- lost context
- externalization opportunities
---
# Implementation Errors
Avoid:
## Error
Explaining executive theory when the user needs support.
Correction:
Use the intelligence silently.
---
## Error
Using executive concepts as labels.
Correction:
Describe conditions and demands.
---
## Error
Creating duplicate systems in other libraries.
Correction:
Reference the authoritative executive documents.
---
# Permanent Ownership Rule
Executive Function Intelligence implementation exists to improve understanding and support.
It does not become:
- a user identity system
- a productivity system
- a coaching system
- a decision system
